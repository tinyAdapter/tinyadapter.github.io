“次元连线”设计思路

发表于 2017-11-13 | 分类于 Web |   |   797 字   |   3 min

最近受Mastodon的启发，有个分布式社交网络的实现想法。

于是首先应该做件事——起个中二的名字。

想来想去，就叫“次元连线”好了（笑）

一个打着泛二次元（二不起，二不起.jpg）幌子的…服务器集群？（

论述一下思路。

首先说明两个基本的大原则：

  1. 用户的文本在本地端加密后再送上服务器；
  2. 服务器是要存储数据的，而不是单纯做转发。

这套体系决定了其可以胜任从数据时间跨度最低的阅后即焚式即时通讯到最高的论坛。

为了叙述省事，先拿论坛举例。

这里把每台服务器（也就是结点）分为三种类型：public, protected和private

（对没错，我就是要强行OOP）

private结点

private最简单，就是不跟其他任何结点相连的单台服务器，自己负责所有的用户注册和数据发送，跟传统的单服务器网站/聊天室本质上没有什么区别。可供私人及小团体使用。

用户请求private服务器后，会得到一个次元群列表。

（次元群：即论坛板块的概念，每个次元群拥有单独的访问密码。对应即时通讯中的群组）

每个结点维护的用户表：

次元群hash 用户id 用户密码hash

每个结点维护的次元群表：

次元群hash 群名 介绍 (群主id+密码hash)的hash【提权】

每个次元群内帖子的信息：

帖子id 标题(使用访问密码加密)

每条跟帖的格式：

次元群hash 帖子id 发表时间 内容(加密)

理论上，每条数据都允许被所有人访问到，但如果不知道密码的话便无法解密（强行访问只会得到一堆乱码）

发表帖子/跟帖时向服务器发送的数据格式：

次元群hash 帖子id 访问密码hash 用户id 用户密码hash 内容(加密)

protected/public结点

protected和public结点类似，都属于内容备份机制，只不过protected结点在信任的若干结点中选取几个做备份，而public选取随机的public结点做备份，同时接受来自任意结点的数据。

而在服务器的组织上，使用Kademlia算法（即DHT）。

每个public结点随机选取3台public结点并作为数据备份结点。

每台public结点维护一个与private结点类似的用户表，其中有本结点内所有次元群的用户id（不同的次元群间允许存在相同的id，以密码hash作为校验手段）

假设某public结点A选择X、Y、Z作为数据备份，则任意结点的更新（某次元群新加入用户、帖子/跟帖）会传播到其余的三个结点。

结点A在建立新的次元群时随机生成一个infohash，按得到的值跳跃到距离hash最近的public结点B，并将infohash，A、X、Y、Z的ip和port存过去。结点B接受结点A的信息后，将其由近及远地广播给附近的public结点。

（完）
