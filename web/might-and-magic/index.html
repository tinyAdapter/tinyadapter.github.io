将魔法隐藏在任意门后

发表于 2017-09-18 | 分类于 Web |   |   1,218 字   |   4 min

emmm

相信每个使用魔法的人对自己的指挥棒都是很看重的（毕竟没有人会想隔三差五来个大搬家），因此怎样保护好自己的指挥棒不让恶龙发现，就变成了一门艺术。

尤其是在普通的ss协议本身存在缺陷的情况下，我们仍不能保证万全。

v2是个好项目，在这里着重感谢一下。

问题

众所周知，一个保护自己指挥棒的方法就是为它加上咒语分发，这样恶龙就没办法偷走指挥棒的编号了。

但是问题来了，咒语分发一般都是只管80/443端口的，80端口直接免谈，而我们在指挥棒上搭个站啊写个博客啊什么的需求又导致了我不能把443端口拱手让给魔法

（当然了你要是财大气粗再买一个指挥棒那当我没说）

而且如果有一个普通的网站（贴上社会主义核心价值观就更好了【逃）能够让人正常访问，那么我再在相同的端口搭上魔法服务就显得更加的隐蔽了。

那么如何实现呢？这其实是个很大的问题。

思考

首先我们一定需要一个前置咒语服务器进行流量转发（如apache或nginx），我自己用的是apache。

而apache在2.x（具体版本号记不住了）之后支持了WebSocket协议的转发，只需要直接写ws://就可以实现。

而v2在某个版本之后（还是记不住orz）也支持了完整的WebSocket协议实现，并且还能使用标准TLS加密流量。

这就很有意思了。如果我们使用WebSocket协议，在外面套上TLS后以某个域名A送给服务器，而apache接到这个请求并解密后能够直接通过WebSocket协议头将这个请求转发给v2的监听端口，由v2拿着数据按照原路返回，就能够实现这样的功能。

而当我们用浏览器以https访问域名A时，apache接到请求，解密后发现是http协议，就返回网站内容。

这样就保证了极高的隐蔽性（因为有加密的存在，所以外人只能看到你在访问一个没什么大碍内容的网站）。

接下来配置v2相关，说几个要注意的地方：

  1. network填ws（客户端和服务器都是）
  2. 服务器端不要开启tls！（因为服务器内部转发的流量已经被apache解密，之前在这里被坑了很久）
  3. 客户端要开启tls
  4. 服务器端端口和apache转发的端口要一致（比如这里就是10000端口）
  5. 建议将向除常用端口以外的其他端口的外部访问全部drop掉，防止被恶龙发现

不放心？再套个咒语分发

到目前为止其实已经做到了相当高的隐蔽性，但是如果你还是不放心，可以使用咒语分发再把实际的编号隐藏起来，这样恶龙就只能看见你访问某个咒语分发的编号，而至于咒语分发把请求转发给了谁，抱歉，纵使它有天大的本事也找不到了（笑）

而咒语分发服务我个人推荐Cloudflare，首先一个就是免费，再就是它跟百度云加速有合作（虽然我个人不待见百度，但还是谢了啊），起码有理由相信它目前不会挂掉。

（当然速度这个我就没办法了，但是重在安全）

总结

自此，我们实现魔法的方式如下：

客户端--(TLS+WS)->咒语分发--(TLS+WS)->服务器apache--(WS)->v2

返回结果同理。

这样的好处在于：

  1. 恶龙只能知道你在访问咒语分发，由于TLS的存在不知道内容
  2. 如果恶龙也来访问相同的域名，那么得到的是一个普通的网站
  3. 由于vmess加密的存在，咒语分发也不知道你访问的内容（百度是个什么样的公司，你懂的）
  4. 只有v2知道你在使用魔法
